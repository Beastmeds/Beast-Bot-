# 🆘 YouTube Downloads - Comprehensive Troubleshooting

## Status: All 6 Download Methods Failed

### What Happened
```
✅ 9 Cookies geladen  (cookies are on server)
🔄 Strategie 1-6 alle fehlgeschlagen
🔄 Invidious instances alle fehlgeschlagen  
❌ Datei zu klein: 0 bytes
```

**Meaning:** YouTube is blocking **all** requests, even with:
- ✅ Valid authentication cookies
- ✅ Real user agents
- ✅ Proper headers
- ✅ Alternative services (Invidious)
- ✅ 7 different extraction methods

---

## 🔍 Diagnosis: What's Really Happening?

### Most Likely Causes (Priority Order):

1. **⚠️ Server IP is blacklisted by YouTube** (HIGH)
   - Symptom: All methods fail instantly
   - Cause: Too many download attempts from same IP
   - Fix: Wait 1-24 hours OR use different IP/VPN

2. **⚠️ Cookies are marked as bot-like** (MEDIUM)
   - Symptom: "Precondition check failed" error
   - Cause: Cookies generated by or used by bots
   - Fix: Generate fresh cookies from real browser

3. **⚠️ Network connectivity issue** (LOW-MEDIUM)
   - Symptom: No response from any service
   - Cause: Server firewall, ISP blocking, DNS issue
   - Fix: Check internet, whitelist YouTube

4. **⚠️ Invidious instances are down** (LOW)
   - Symptom: Invidious strategies fail
   - Cause: Invidious is also frequently blocked
   - Fix: Wait for instance recovery

## 🔧 Diagnostic Steps (In Order)

### Step 1: Check Server Connectivity (CRITICAL)
Run this **on the server**:
```bash
# SSH into server, then:
bash diagnose-server-downloads.sh
```

This will tell you:
- ✅ Is Node.js/ffmpeg/yt-dlp installed?
- ✅ Can server reach YouTube?
- ✅ Can server reach Invidious?
- ✅ Does yt-dlp work at all?
- ✅ Are cookies present?

**What to look for:**
```
✅ Can reach youtube.com
✅ $inst is reachable
✅ File created: XXX bytes
→ All green = server is fine
```

If you see "❌ Cannot reach youtube.com" → **Server has no internet**

### Step 2: Manual yt-dlp Test (if diagnostic is OK)
```bash
# On server, test yt-dlp directly:
yt-dlp --cookies /root/Beast-Bot-/youtube/cookies.txt \
  -f best \
  -o /tmp/test.mp4 \
  "https://youtube.com/watch?v=IxX_QHay02M"
```

**Possible results:**
- ✅ File downloads → Server can download, bot code issue
- ❌ "Precondition check failed" → YouTube blocking this IP
- ❌ "Connection timeout" → Network issue

### Step 3: Fresh Cookies (if IP blocking suspected)
```bash
# On LOCAL machine (your computer), generate new cookies:
yt-dlp --cookies-from-browser chrome https://www.youtube.com

# Then sync:
bash sync-cookies.sh
```

**Why this helps:**
- Old cookies might be marked as bot-like
- Chrome cookies might be trusted better than Firefox
- Fresh generation = fresh session

### Step 4: Check IP Reputation (if still failing)
```bash
# On server:
curl -s https://checkip.amazonaws.com  # Shows your IP
```

Then check if IP is blacklisted:
- Visit: https://www.abuseipdb.com/
- Search your IP
- High score = blacklisted by YouTube

**If blacklisted:**
- Wait 24-48 hours for YouTube to unblock
- OR use VPN/proxy to change IP
- OR whitelist IP with YouTube (difficult)

## 💡 Solutions by Severity

### 🔴 CRITICAL: Server Has No Internet
```bash
# Fix network issue first
ping 8.8.8.8
# If no response = no internet
# Check: firewall, DNS, ISP, network cable
```

### 🟠 HIGH: YouTube Blocks This IP
**Options:**
1. **Wait 12-24 hours** - YouTube auto-unblocks after cooldown
2. **Use VPN on server** - Change IP address
3. **Switch server** - Use different cloud provider
4. **Ask YouTube for unblock** - (very unlikely to help)

**Temporary workaround:**
- Use `/play` for audio only (sometimes less detected)
- Wait between multiple downloads
- Tell users YouTube is blocking

### 🟡 MEDIUM: Cookies Marked as Bot
**Fix:**
```bash
# 1. Generate fresh cookies (on local machine)
yt-dlp --cookies-from-browser firefox https://www.youtube.com
# Try different browser:
#   firefox, chrome, edge, safari

# 2. Sync to server
bash sync-cookies.sh

# 3. Restart bot
pm2 restart BeastBot

# 4. Test
/mp4 https://youtube.com/watch?v=<VIDEO_ID>
```

### 🟢 LOW: Implementation Bug
If manual yt-dlp works but `/mp4` doesn't:
- Check bot logs for errors
- Verify paths are correct
- Check file permissions
